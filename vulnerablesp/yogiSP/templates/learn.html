Learn About SAML

This application is designed to demonstrate pitfalls when configuring applications to use SAML. While SAML can be vulnerable when misconfigured, there is nothing inherently wrong with it when implemented properly.

Key Terminology
TermDescription
Identity Provider (IDP) The server that handles authentication. In this environment, the IDP is the "Jellystone IDP." This is who you prove your identity to. An IDP may connect to AD or another user database, and can help easily implement multi-factor authentication.
Service Provider (SP) The application a user is trying to access. In this case, the SP is "Yogi's SAML App."
Client That's you. A client is anyone or anything sending a request to the SP.
Authentication Flows

The most important thing is ensuring that a service provider checks that messages from the Identity Provider are signed and valid.

If messages are not signed, or are signed but not validated, the application should reject them. An attacker could:

  • Remove the signature and edit attributes
  • Leave the signature and edit attributes
  • Edit the attributes and re-sign with their own key

Each of those actions should result in a login failure.
